# Design V5.1 — Path Resolver do vault-pilot

## 1. Contexto

O agente `vault-pilot` depende de um resolver de path para transformar pedidos do usuário do tipo
“analise o contrato do Conrado” em um path concreto dentro do vault (ex. `CONRADO/GERAL/Contrato Conrado Nogueira.pdf`).

Nos experimentos V1–V4, ficou claro que o modelo se comporta bem quando recebe `<file_content>` correto,
e que problemas de análise decorrem principalmente de falhas em ferramentas (listagem, leitura, path resolver).

O V5 — Fuzzy Path Resolver Benchmark mostrou que:

- O resolver atual baseado em regex (`_extract_filename_intent` + `_extract_query_name`) falha em 0/10 casos simples.
- A regex `[^\\s]+\\.pdf` captura apenas a última “palavra.pdf”, por exemplo:
  - `"CONRADO/GERAL/Contrato Conrado Nogueira.pdf"` → `"Nogueira.pdf"`.
- Em um vault com muitos arquivos contendo “Conrado”/“Nogueira”, isso gera ambiguidade sistemática.
- O novo `_extract_filename_intent_fuzzy` acerta alguns casos (4/10, sendo 3 reais + 1 acidental), mas:
  - depende fortemente de o `query_name` ser informado,
  - ainda não é robusto o suficiente para produção.

O objetivo do V5.1 é corrigir a extração de nomes de arquivo e refinar as regras do fuzzy resolver, mantendo tudo
ainda em modo experimental, mas mais próximo de um candidato a produção.

---

## 2. Objetivos do V5.1

### 2.1. Objetivo funcional

- Permitir que o agente resolva, com alta precisão, o arquivo solicitado pelo usuário a partir de inputs naturais
  (paths completos, nomes de arquivo com espaços, frases, apelidos), ou declare ambiguidade/ausência de forma previsível.

### 2.2. Objetivo de qualidade

- Aumentar a acurácia do resolver fuzzy em um dataset maior (40–60 casos) em relação ao V5, com foco em:
  - reduzir falsos negativos (não encontrar arquivos existentes),
  - evitar falsos positivos perigosos (apontar um arquivo errado em vez de marcar “none/ambiguous”).

---

## 3. Escopo

Inclui:

- Refino da função de extração de query (nome do arquivo) a partir do utterance:
  - `_extract_query_name`.
- Refino da função `_extract_filename_intent_fuzzy`, incluindo:
  - normalização,
  - scoring,
  - regras de decisão (exact/fuzzy/ambiguous/none).
- Atualização do script de benchmark:
  - `scripts/run_v5_path_benchmark.py` (dataset ampliado, métricas adicionais).

Não inclui (neste V5.1):

- Alterar o comportamento de produção do `vault-pilot`.
- Ativar automaticamente o fuzzy resolver em runtime.
- Refatorar outros componentes do engine (compressão, injeção, etc.).

---

## 4. Design de `_extract_query_name` (V5.1)

### 4.1. Problema atual

- Implementação atual usa regex do tipo `[^\\s]+\\.pdf` (e similares) para detectar nomes de arquivos.
- Isso captura apenas a última “palavra” antes de `.pdf`:
  - `"CONRADO/GERAL/Contrato Conrado Nogueira.pdf"` → `"Nogueira.pdf"`.
- Em vaults reais com:
  - nomes compostos,
  - diretórios aninhados,
  - múltiplos arquivos por pessoa,
  essa extração é insuficiente e produz ambiguidades em série.

### 4.2. Nova abordagem em camadas

A função `_extract_query_name(user_utterance: str, file_paths: list[str])` deve seguir um pipeline:

#### Camada 1 — Match de path completo

- Objetivo: se o usuário colar o path completo (ou quase completo), usar isso diretamente.
- Estratégia:
  - Normalizar `user_utterance` e cada `file_path`:
    - lower-case,
    - remoção de acentos,
    - normalização de separadores (`/` vs `\`),
    - compressão de espaços.
  - Se um `file_path` normalizado aparecer como substring no `user_utterance` normalizado, retornar:

    ```python
    return {
        "query_name": None,
        "match_hint_path": file_path,
        "tokens": None,
        "extraction_source": "full_path_substring",
    }
    ```

- Motivo: evita passar pelo fuzzy quando o match é trivial.

#### Camada 2 — Extração de basename com extensão (aspas / padrão)

- Procurar primeiro texto entre aspas (`"..."` ou `'...'`):

  - Se encontrar:
    - usar a string inteira entre aspas como candidata (`candidate_name`),
    - não dividir por espaços.

- Se não houver aspas, procurar padrões com extensão:

  - Regex mais robusta, por exemplo:
    - capturar a sequência de caracteres (incluindo espaços) até a extensão, não só a última palavra.
  - Ex.: usar um padrão que permita `.pdf/.doc/.docx/.xlsx` com espaços antes da extensão.

- Resultado esperado:

  ```python
  return {
      "query_name": candidate_name,  # ex.: "Contrato Conrado Nogueira.pdf"
      "match_hint_path": None,
      "tokens": None,
      "extraction_source": "quoted_or_extension",
  }
  ```

#### Camada 3 — Fallback baseado em tokens fortes

- Se não houver aspas nem extensão clara:

  - Tokenizar `user_utterance`:
    - remover stopwords (português),
    - focar em nomes próprios/sobrenomes/termos específicos: "contrato", "procuração", "rg", "rg emilia", etc.
  - Montar:
    - um `query_name` textual (ex.: `"contrato conrado nogueira"`), e/ou
    - um conjunto de tokens fortes, ex.:

      ```python
      tokens = ["contrato", "conrado", "nogueira"]
      ```

- Resultado:

  ```python
  return {
      "query_name": "contrato conrado nogueira",
      "match_hint_path": None,
      "tokens": tokens,
      "extraction_source": "tokens_fallback",
  }
  ```

### 4.3. Normalização

- Função auxiliar consistente com o V5:

  - NFKD,
  - remoção de acentos,
  - lower-case,
  - remoção de caracteres não alfanuméricos relevantes (exceto `/`, `.`, `_`, `-`, espaço),
  - compressão de múltiplos espaços.

Essa normalização deve ser usada tanto em `query_name` quanto em `basename`/`file_path`.

## 5. Design de `_extract_filename_intent_fuzzy` (V5.1)

### 5.1. Entradas e saídas

Assinatura:

```python
def _extract_filename_intent_fuzzy(
    user_utterance: str,
    file_paths: list[str],
) -> dict:
    """
    Retorna um dict com:
      - "match_type": "exact" | "fuzzy" | "ambiguous" | "none"
      - "resolved_path": str | None
      - "candidates": list[{"path": str, "score": float, "reason": str}]
      - "extraction": resultado de _extract_query_name (para debug)
    """
```

### 5.2. Integração com `_extract_query_name`

- Primeiro passo: chamar `_extract_query_name` e obter:

  - `match_hint_path` (se Camada 1 acionar),
  - `query_name`,
  - `tokens`,
  - `extraction_source`.

- Se `match_hint_path` não for `None`:
  - Isso deve ser tratado como candidato de score máximo (próximo de um match exato).

### 5.3. Scoring

- Para cada `file_path` em `file_paths`:

  - Calcular:
    - `basename` normalizado,
    - opcionalmente, `directory` normalizado.

- Sinais de score:

  - Similaridade de string:
    - `SequenceMatcher` ou equivalente entre `query_name_norm` e `basename_norm`.
  - Overlap de tokens:
    - se tiver `tokens` fortes, calcular proporção de tokens de `tokens` presentes em `basename_norm`.
  - Sinal de substring:
    - `query_name_norm in basename_norm` ou vice-versa.

- Score composto (exemplo):

  - `score = base_score + alpha * overlap + beta * substring_hit`
  - Limitar `score` a [0.0, 1.0].

### 5.4. Regras de decisão

Após ordenar candidatos por `score`:

- `exact`:

  - Se:
    - `match_hint_path` estiver presente e corresponder a um path real → pode ser tratado como `exact`.
    - OU:
      - `score_top1 == 1.0`
      - e nenhum outro candidato tiver `score == 1.0`.
  - `resolved_path = path_top1`, `match_type = "exact"`.

- `fuzzy`:

  - Se:
    - `score_top1 >= SCORE_FUZZY_MIN` (por exemplo, 0.8),
    - `delta = score_top1 - score_top2 >= DELTA_MIN` (por exemplo, 0.15),
    - tokens fortes (se existirem) estão todos presentes em `basename_norm`.
  - `match_type = "fuzzy"`.

- `ambiguous`:

  - Se:
    - `score_top1 >= SCORE_AMBIGUOUS_MIN` (por exemplo, 0.5),
    - mas `delta < DELTA_MIN` (scores muito próximos),
    - ou vários candidatos contêm o mesmo conjunto de tokens fortes.
  - `match_type = "ambiguous"`,
  - `resolved_path = None`,
  - `candidates = top-3` para debug.

- `none`:

  - Se:
    - `score_top1 < SCORE_AMBIGUOUS_MIN`, ou
    - nenhum candidato satisfaz as regras de tokens/substrings.
  - `match_type = "none"`,
  - `resolved_path = None`.

### 5.5. Casos especiais (como emilia_rg)

- Para queries com tokens compostos (ex.: `"emilia rg"`):

  - Exigir que todos os tokens fortes estejam presentes nos candidatos:
    - se nenhum arquivo satisfez “emilia” AND “rg” → `match_type = "none"`.
  - Isso evita escolher arbitrariamente um arquivo com “emilia” qualquer.

## 6. Benchmark V5.1 — Plano

### 6.1. Dataset

- Selecionar 3–5 “pessoas” ou entidades do vault (ex.: CONRADO, EMILIA, outras pastas relevantes).
- Para cada uma, escolher 3–5 arquivos com nomes representativos:
  - contratos, procurações, relatórios, RG/CPF/dados pessoais, etc.

Para cada arquivo, gerar queries:

- Path completo:
  - `"CONRADO/GERAL/Contrato Conrado Nogueira.pdf"`.
- Basename com extensão:
  - `"Contrato Conrado Nogueira.pdf"`.
- Basename sem extensão:
  - `"Contrato Conrado Nogueira"`.
- Aspas simples/duplas:
  - `"Contrato Conrado Nogueira.pdf"`, `'Contrato Conrado Nogueira.pdf'`.
- Frase natural longa:
  - `"Use as ferramentas para analisar o contrato do Conrado Nogueira que está na pasta geral."`.
- Apelidos/abreviações:
  - `"analise o contrato do Conrado"`,
  - `"procuração da Emilia"`, etc.
- Casos confusos:
  - `"relatório nogueira"` quando existem vários relatórios com esse sobrenome.

Casos negativos:

- Queries com nomes inexistentes ou combinações impossíveis:
  - `"contrato joao nogueira 2010"` (sem arquivo correspondente),
  - `"emilia cpf contrato 2010"` se não houver.

Tamanho alvo:

- 40–60 queries no total:
  - 5 pessoas × 5 arquivos × 2–3 queries cada, + ~10–15 negativos.

### 6.2. Script

- Atualizar `scripts/run_v5_path_benchmark.py` para:

  - Incluir o novo dataset de casos.
  - Registrar, para cada query:
    - `expected_path`,
    - resultado do resolver atual (regex),
    - resultado do resolver fuzzy (V5.1),
    - campos auxiliares: `match_type`, `resolved_path`, `candidates`, `extraction_source`.

- Salvar resultados em:

  - `results/benchmarks/V5/` com timestamp (sem sobrescrever o JSON de 20260427-1316).

### 6.3. Métricas e critérios de sucesso

- Para o fuzzy (V5.1):

  - Acurácia:
    - proporção de queries com `resolved_path == expected_path`.
  - Ambiguidade:
    - % de queries marcadas como `ambiguous`.
  - Falsos negativos:
    - % de queries com arquivo existente, mas `match_type = "none"`.
  - Falsos positivos:
    - qualquer query onde `resolved_path != expected_path` e `expected_path` não é `None`.

Critério (ajustável):

- Acurácia ≥ 80–90% em queries “bem especificadas” (basename, path, aspas, nome + sobrenome).
- Nenhum falso positivo em casos negativos, ou um número muito baixo, com análise manual.
- Ambiguidades apenas em queries realmente ambíguas (múltiplos arquivos quase idênticos).

---

## 7. Riscos e limitações

- Vault altamente redundante:
  - Muitos arquivos com mesmo sobrenome podem manter alta taxa de ambiguidades; isso é inerente aos dados.
- Generalização:
  - Heurísticas afinadas para este vault podem não generalizar para outros; por isso, V5.1 deve ser tratado como
    “resolver do vault-pilot” e não necessariamente global.
- Rollout:
  - V5.1 não será ativado em produção automaticamente:
    - será avaliado com V5.1, documentado, e só então considerado para ser ligado com feature flag.
