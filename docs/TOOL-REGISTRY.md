# Tool Registry — Agents that Build Tools

## Overview

The **Tool Registry** is the mechanism by which AgentForge extends itself: agents create new Python tools at runtime, test them with pytest, and register them permanently in the framework. The next time any agent initializes, the registered tools will be available as if they were builtins.

This implements the **Voyager** pattern (NVIDIA, 2023) for local agents: progressive accumulation of skills generated by the system itself.

---

## Architecture

```
agents-framework/
├── tool_registry/                 ← Tools generated by agents
│   ├── registry.yaml              ← Persistent manifest
│   └── search_memory.py           ← Example: created by tool-builder
│
├── src/agentforge/tools/
│   ├── register_tool_file.py      ← Builtin tool: registers files in the registry
│   ├── dynamic_loader.py          ← Loads registry.yaml at initialization
│   └── registry.py                ← Main registry (builtins + dynamic)
│
└── agents/tool-builder/           ← Agent specialized in creating tools
    └── agent.yaml
```

### Creation flow

```
1. tool-builder receives tool description
        ↓
2. Writes <tool>.py with write_file
        ↓
3. Reads implementation with read_file (coherence check)
        ↓
4. Writes test_<tool>.py with write_file
        ↓
5. Runs pytest with run_bash (real validation)
        ↓
6. If tests pass: calls register_tool_file
        ↓
7. tool_registry/<tool>.py + registry.yaml updated
        ↓
8. Tool available immediately (current process) and in every future session
```

### Automatic loading

At initialization, `_register_builtin_tools()` in `registry.py` calls `load_dynamic_tools()`, which reads `registry.yaml` and imports each Python file dynamically. Registered tools have the same status as builtins — any agent can use them by declaring the name in `agent.yaml`.

```python
# src/agentforge/tools/dynamic_loader.py
def load_dynamic_tools() -> list[str]:
    """Loads tools registered in tool_registry/registry.yaml into the live _ToolRegistry."""
```

## registry.yaml

Persistent manifest of all registered tools. Do not edit manually — managed by `register_tool_file`.

```yaml
tools:
  - name: search_memory
    file: search_memory.py
    function: search_memory
    description: Busca na agent-mesh shared_memory por query LIKE no key e value.
    input_schema: '{"type":"object","properties":{"query":{"type":"string"},"db_path":{"type":"string"}},"required":["query"]}'
    created_by: tool-builder
    registered_at: '2026-06-06T20:14:00.325922+00:00'
```

Fields:

| Field | Description |
|---|---|
| `name` | Tool name — used in `agent.yaml` and in `execute_tool` |
| `file` | Python filename inside `tool_registry/` |
| `function` | Name of the public function to expose |
| `description` | Description used in the tools schema (context for the model) |
| `input_schema` | JSON schema of parameters (JSON string) |
| `created_by` | Identifier of the creating agent |
| `registered_at` | ISO 8601 UTC |

## register_tool_file (builtin)

Builtin tool available to any agent. Validates, copies, and registers a Python file in the registry.

**Parameters:**

```python
def register_tool_file(
    source_path: str,      # Path relative to AGENT_WORKDIR
    tool_name: str,        # Unique snake_case name in the registry
    function_name: str,    # Name of the public function to expose
    description: str,      # Description for the tools schema
    input_schema: str,     # JSON schema as string (default: "{}")
    created_by: str,       # Creator identifier (default: "tool-builder")
) -> dict
```

**Return on success:**

```json
{
  "success": true,
  "tool_name": "search_memory",
  "registry_path": "/home/user/agents-framework/tool_registry/search_memory.py",
  "message": "Tool 'search_memory' registrada com sucesso em tool_registry/"
}
```

**Validations run before registering:**

1. File exists in `AGENT_WORKDIR`
2. Valid Python syntax (`ast.parse`)
3. Public function exists in the module (`getattr`)
4. Module imports without errors

If any validation fails, returns `{"success": false, "error": "<reason>"}` without touching the registry.

## tool-builder Agent

Agent specialized in creating and registering tools. Located in `agents/tool-builder/`.

**Model:** `qwen3.5:27b` (default — multi-step coding tasks)

**Available tools:** `write_file`, `read_file`, `run_bash`, `register_tool_file`

**Guardrails:**
- `must`: re-read the implementation before writing tests; run pytest before registering; register only with passing tests
- `must_not`: register with failing tests; claim that pytest passed when it did not

**CLI usage:**

```bash
agentforge run --agent-dir agents/tool-builder --input "
Crie uma tool Python chamada rate_limiter que:
- Recebe (key: str, limit: int, window_s: int) -> bool
- Usa Redis local para controle de janela deslizante
- Testa com pytest usando Redis mockado
- Registra no framework após testes passarem
"
```

**Programmatic usage:**

```python
from agentforge.runtime.engine import AgentRuntime
import os

os.environ["AGENT_WORKDIR"] = "/tmp/my_tool_build"
runtime = AgentRuntime.from_agent_dir("agents/tool-builder")
result = runtime.run("Crie uma tool ...")
```

## Using registered tools in other agents

Once registered, the tool appears in `_ToolRegistry` under its `tool_name`. To use it in any agent, declare it in `agent.yaml`:

```yaml
tools:
  - name: search_memory
    description: Busca na agent-mesh shared_memory por query LIKE no key e value.
    when_to_use: "Usar quando precisar recuperar contexto de sessões anteriores."
    input_schema: '{"type":"object","properties":{"query":{"type":"string"}},"required":["query"]}'
```

The engine generates the correct schema for Ollama — the model decides when to call it.

## Limitations and next steps

### Current limitations

- The model (`qwen3.5:27b`) completes ~80% of the flow autonomously. The remaining 20% (schema inconsistencies between separate files) requires human review or fine-tuning.
- Each `register_tool_file` overwrites the previous version of the tool — there is no versioning.
- Dynamic tools are not validated on every startup — only loaded. If the `tool_registry/*.py` file is corrupted, the error is silent.

### Next steps

- **Fine-tuning**: use `docs/FINETUNING-STRATEGY.md` to train qwen3.5 to complete 100% of the flow without intervention
- **Versioning**: `registry.yaml` to store version history (simple semver)
- **Startup validation**: `load_dynamic_tools` to check syntax before registering
- **Tool discovery**: CLI `agentforge tools list` to list builtins + dynamic tools
